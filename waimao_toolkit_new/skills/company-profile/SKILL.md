# Company Profile Skill — 企业能力档案构建

本 Skill 是整套客户开发系统的**素材底座**。它将公司零散的销售资料整理成 AI 可直接调用的结构化档案，后续开发信、客户匹配等 Skill 都依赖这里的输出。

**核心理念**：让 AI 真正理解"我们是谁、我们卖什么、我们能帮客户什么"，而不是每次从零开始编。

**Skill 路径**：`__SKILL_DIR__/company-profile/`

## 工作流程概览

```
Phase 0: 检查已有档案（是否存在 profile.json）
    ↓
Phase 1: 引导收集信息（友好分步对话）
    ↓
Phase 2: 资料采集（官网抓取 + 文档读取）
    ↓
Phase 3: AI 深度整理（结构化分析）
    ↓
Phase 4: 输出档案（profile.json + profile.md）
```

---

## Phase 0: 检查已有档案

开始前先检查是否已有档案：

```
__SKILL_DIR__/company-profile/profile.json
```

- **如果存在**：读取现有档案，告知用户"已找到上次保存的档案，以下是现有内容摘要... 您想更新哪些部分？"进入**更新模式**（只问变化的部分）
- **如果不存在**：进入**全新创建模式**，从头开始引导

## Phase 1: 引导收集信息

**设计原则**：不要一次性丢出一大堆问题。像一位经验丰富的销售顾问一样，自然地一步步引导。

### 1a. 开场白

全新创建时的开场：

```
我来帮您建立公司的销售能力档案。这份档案是后续所有开发信和客户匹配的基础，
档案越完整，AI 生成的开发信就越精准、越有针对性。

我们先从最基础的信息开始。请告诉我：
1. 公司全称是什么？
2. 公司在哪个城市？
3. 主要做什么行业/产品？
```

更新模式时的开场：

```
已找到您上次保存的档案（{公司名}，更新于 {日期}）。

以下是当前档案的摘要：
- 主营产品：{产品列表}
- 核心卖点：{卖点列表}
- 适合客户：{客户类型}

您想更新哪些部分？可以直接告诉我变化的内容，也可以说"全部更新"重新来过。
```

### 1b. 分步引导（全新创建模式）

根据用户回答，逐步深入。每次只问 1-2 个相关问题，根据用户回答的内容灵活调整后续问题：

**第一轮：基础信息**（必须收集）
- 公司名称、所在地、行业、成立时间、公司规模（人数/年产值）

**第二轮：核心业务**（必须收集）
- 主营产品/服务列表（每类产品简单说明）
- 公司官网 URL（如果有）

> **重要：收到官网 URL 后立即抓取**
>
> AI agent 通常无法直接访问网页。用户提供了官网 URL 后，**必须立即**使用 Bash 工具运行 Playwright 抓取脚本：
>
> ```bash
> cd __SKILL_DIR__/company-profile/scripts
> python scrape_website.py --url "用户提供的URL" --output "__SKILL_DIR__/company-profile/sources/website_raw.md"
> ```
>
> 抓取完成后：
> 1. 用 Read 工具读取 `sources/website_raw.md` 的内容
> 2. 基于抓取结果向用户确认/补充信息（如："我从官网看到贵司主要做XXX，还有其他产品线吗？"）
> 3. 抓取失败时（依赖缺失等），提示用户安装依赖或手动提供信息

**第三轮：竞争优势**（尽量收集）
- 与同行相比，最大的优势是什么？
- 有哪些证书/资质/认证？
- 有没有工厂/展厅/办公场所？（影响信任背书）

**第四轮：客户与案例**（尽量收集）
- 目前主要服务什么类型的客户？（品牌商/批发商/分销商/工程项目/跨境电商等）
- 有没有比较典型的成功案例？
- 通常的合作模式是什么？（OEM/ODM/经销/代理等）

**第五轮：补充信息**（可选）
- 目标市场/想开发哪些国家的客户？
- 有没有产品目录/画册/PDF 可以提供？
- 有没有飞书文档链接可以参考？
- 还有什么想补充的？

**关键交互技巧**：
- 用户回答模糊时，追问具体例子。比如用户说"质量好"，追问"能举个具体例子吗？比如用了什么材料、通过了什么检测？"
- 用户提到案例时，追问"这个客户是什么类型的？合作前后有什么变化？"
- 如果用户只提供了官网 URL，先去抓取，回来后再基于抓取结果做针对性补充
- 每轮结束后简短确认："好的，已记录。接下来..."

## Phase 2: 资料采集

### 2a. 官网抓取（用户提供了 URL 时必须执行）

> 如果在 Phase 1 已经抓取过，直接使用已有的 `sources/website_raw.md`，跳到 Phase 2a-2 案例深度挖掘。
> 如果 Phase 1 尚未抓取（如用户在后续对话中才补充 URL），在此处执行抓取。

使用 Playwright 脚本抓取官网：

**前置检查**：先确认抓取依赖可用：

```bash
python -c "import playwright; print('OK')" 2>/dev/null
```

如果失败，先安装依赖：

```bash
pip install playwright --quiet && python -m playwright install chromium
```

然后执行抓取：

```bash
cd __SKILL_DIR__/company-profile/scripts
python scrape_website.py --url "https://example.com" --output "__SKILL_DIR__/company-profile/sources/website_raw.md"
```

脚本会自动：
1. 访问首页，提取页面文本和导航链接
2. 识别并访问 About/Company、Products/Services、Case Studies/Projects、Contact 等关键页面
3. 将所有抓取内容整合为一个 Markdown 文件

抓取完成后，将结果保存到 `sources/` 目录作为原始资料备份。

### 2a-2. 案例深度挖掘（关键步骤）

官网抓取完成后，**必须**进行案例深度挖掘。这是后续开发信素材的核心来源。

**操作步骤**：

1. **识别案例列表**：从抓取结果中找到 Case Studies / Projects / Project Gallery 等页面，检查是否有项目列表
2. **逐个访问项目详情页**：对于列表中的每个项目，尝试访问其详情页
   - 使用 `scrape_website.py` 的 `--pages cases` 选项，或手动指定案例详情页 URL
   - 如果案例详情页较多（超过 10 个），优先选择包含具体数据的页面（有面积、数量、行业描述的优先）
3. **提取关键信息**：对于每个项目案例，必须提取以下维度：
   - 具体产品类型（如"铝合金吊顶板 600×600mm"而非笼统的"吊顶材料"）
   - 面积/数量/规模（如"15,000 ㎡"、"5,000 套"）
   - 客户行业背景（如"迪拜五星级酒店"、"美国商业综合体"）
   - 项目挑战（客户遇到了什么问题）
   - 解决方案（我们怎么解决的）
   - 交付结果（可量化的成果）

4. **信息不足时主动补充**：
   - 如果官网项目详情页内容太简单（SPA 页面无法加载、或只有图片没有文字），尝试用 `WebSearch`（如果当前环境可用）搜索补充信息，如搜索 `"{公司名} {项目名} case study"` 或 `"{公司名} {行业} project"`
   - `WebSearch` 是**可选增强**，不是必需依赖。如果不可用，记录 `[待补充]` 并在档案中标注
5. **案例数量目标**：至少收集 **10 个**有实质内容的案例，上不封顶

### 2b. 文档读取（如果用户提供了飞书文档/PDF等）

- **飞书文档**：使用 `lark-doc` skill 读取文档内容
- **本地 PDF**：使用 Read 工具读取（支持 PDF）
- **本地文件**：使用 Read 工具读取

读取的内容也保存到 `sources/` 目录。

### 2c. 整合所有原始资料

将以下信息整合为"原始资料包"：
- 官网抓取内容（如果有）
- 用户直接输入的文字信息
- 外部文档内容（如果有）

## Phase 3: AI 深度整理

这是最关键的一步。拿到原始资料后，AI 需要像一个资深销售经理一样，深入理解这家公司。

### 分析原则

1. **不只做简介，要构建销售认知**：每一条信息都要考虑"这怎么用在开发信里"
2. **具体化**：把"质量好"转化为"通过 ISO 9001 认证，10 年零退货记录"
3. **客户视角**：不只说我们有什么，要说明这对客户意味着什么
4. **诚实标注**：不确定的信息标注 [待确认]，没有的信息留空而不是编造
5. **边界意识**：明确哪些是我们能说的，哪些是不能乱说的

### 输出结构

完整的 JSON 结构定义见 `references/json-schema.md`。核心字段说明：

| 字段 | 用途 | 后续谁用 |
|------|------|----------|
| `company_name` | 公司名称 | 所有 Skill |
| `one_line_intro` | 一句话介绍 | 开发信开头 |
| `full_intro` | 完整介绍 | 公司介绍段落 |
| `products[].target_customers` | 每类产品的目标客户 | 客户匹配 |
| `core_competencies[]` | 核心竞争力 | 开发信卖点 |
| `target_customer_types[]` | 适合开发的客户类型 | 客户筛选 |
| `case_studies[]` | 成功案例 | 开发信信任背书 |
| `unique_selling_points[]` | 独特卖点 | 开发信核心论点 |
| `customer_matching_guide[]` | 客户匹配建议 | 开发信个性化 |
| `boundaries` | 信息边界 | 防止 AI 乱写 |

### 案例分析标准（Phase 3 核心）

每个案例**必须**包含以下完整字段，不允许只写一句话了事：

| 字段 | 说明 | 示例 |
|------|------|------|
| `project` | 项目/案例名称（中文） | "迪拜五星级酒店吊顶项目" |
| `project_en` | 英文项目名 | "Dubai 5-Star Hotel Ceiling Project" |
| `client_type` | 客户类型 | "品牌酒店集团" |
| `industry` | 客户所属行业 | "酒店/餐饮" |
| `country` | 国家/地区 | "阿联酋（迪拜）" |
| `products_used` | 具体使用的产品 | `["铝合金吊顶板 600×600mm", "铝蜂窝板"]` |
| `area_or_quantity` | 面积/数量/规模 | "15,000 ㎡" |
| `problem_solved` | 解决了什么问题 | "需要防火A级且造型复杂的异形吊顶" |
| `result` | 交付结果/客户反馈 | "45天完成交付，通过当地消防检测" |
| `key_highlight` | 一句话精华（≤30字） | "45天交付15,000㎡防火吊顶" |
| `usable_in_outreach` | 是否可用于开发信 | `true` |

**`key_highlight` 的设计目的**：专门为开发信使用的一句话亮点。要求：
- 包含具体数字（面积、数量、时间等）
- 突出核心优势（速度、规模、质量等）
- ≤30 个字，可直接嵌入开发信
- 例："30天交付20,000㎡外墙铝单板"、"为500强企业提供OEM定制15年"

**案例分类与筛选**：
- 按行业分类（酒店、商业综合体、住宅、公共建筑等）
- 按地区分类（中东、东南亚、北美、欧洲等）
- 同行业/同地区案例只保留**最有代表性**的 2-3 个
- 优先保留有具体数字和明确结果的案例
- 案例数量目标：**至少 10 个**，如果资料充足则尽量多收集

### 特别注意：boundaries 字段

这个字段非常重要，它防止后续开发信 Skill 乱写：

```json
"boundaries": {
  "claims_we_can_make": [
    "通过 ISO 9001 认证",
    "15 年行业经验",
    "月产能 50000 件"
  ],
  "claims_we_cannot_make": [
    "不能声称是世界第一/最大",
    "不能承诺具体价格（需报价）",
    "不能编造不存在的案例"
  ],
  "sensitive_topics": [
    "不主动提及竞争对手",
    "不讨论最低起订量（除非用户确认）",
    "不提及具体客户名称（除非获得授权）"
  ]
}
```

**路径**：`__SKILL_DIR__/company-profile/profile.json`

### 4c. 结束语

档案生成后，向用户展示摘要并告知后续用途：

```
企业能力档案已生成并保存！

档案摘要：
- 公司名称：{name}
- 主营产品：{products 概要}
- 核心卖点：{USP 概要}
- 适合客户类型：{types}

文件位置：
- 结构化数据：profile.json
- 可读版本：profile.md
- 原始资料：sources/

这份档案将作为后续开发信生成和客户匹配的基础数据。
您可以随时回来更新任何部分——只需告诉我"更新公司档案"即可。
```

> **配置 FROM_EMAIL（发件人邮箱）**
>
> 档案建好后，如果 `.env` 中的 `FROM_EMAIL` 还是占位符，**现在**可以配置了：
>
> - 从档案中读取公司名/品牌名作为 FROM_EMAIL 的品牌部分
> - 格式：`{公司名/品牌名} <sales@{已验证域名}>`
> - **禁止**用域名自动拼接品牌名——域名≠公司名
> - 写入命令：
>   ```bash
>   python __SKILL_DIR__/email-blast/scripts/run.py --setup FROM_EMAIL="{品牌名} <sales@域名>"
>   ```
> - 域名必须在 Resend 完成 DNS 验证后才能发邮件。如果尚未验证，先跳过，等发送前再配置。

## 更新模式

当用户回来更新档案时：
1. 读取现有 profile.json
2. 展示当前摘要
3. 询问要更新哪些部分
4. 只重新处理变化的部分，保留未变化的内容
5. 更新 metadata.updated_at
6. 重新输出 profile.json 和 profile.md

本 Skill 的输出是以下 Skill 的输入：

| 下游 Skill | 使用方式 |
|------------|----------|
| customer-acquisition | 读取 `my-company` 和 `my-products` 做客户匹配 |

## 技术依赖

### 基础依赖（开箱即用）
- **Python 3.8+**
- **requests**：`pip install requests`
- **beautifulsoup4**：`pip install beautifulsoup4`

> 脚本 `scripts/scrape_website.py` 会自动检测依赖状态。如果 Playwright 不可用，自动降级到 requests + beautifulsoup4，仍可抓取静态页面内容。

### 推荐依赖（增强抓取能力）
- **Playwright**（SPA 网站需要）：`pip install playwright && python -m playwright install chromium`
  - 适用于需要 JavaScript 渲染的现代网站（SPA、React/Vue 等框架构建的站点）
  - 静态页面（传统 HTML 网站）不需要此依赖

### 环境增强（有则更好，无则不影响核心流程）
- **WebSearch**（可选增强）：如果当前运行环境提供 WebSearch 工具，可在案例深度挖掘时搜索补充信息（如 `"{公司名} {项目名} case study"`）。如果不可用，记录 `[待补充]` 即可
- **webReader MCP**（可选增强）：如果已配置 MCP server，可作为网页抓取的备选方案
- **飞书 CLI**（读取飞书文档）：`npm install -g @larksuite/cli`

### 依赖检测与降级机制

`scripts/scrape_website.py` 启动时会自动检测并打印依赖状态：

```
[依赖状态] Playwright: ✓ 可用          → 使用浏览器模式（支持 SPA）
[依赖状态] Playwright: ✗ 不可用        → 降级到 requests 模式（仅静态页面）
[依赖状态] requests+bs4: ✓ 可用        → 降级模式可用

降级模式下仍能完成核心抓取任务，只是无法处理 JavaScript 渲染的页面。对于大多数企业官网（非 SPA），降级模式完全够用。
